# Development Log: March 30, 2026

## Goal

Document the next major product direction for MacroLens, with explicit focus on fixing the weak real-world context layer.

## Work Completed

- reviewed the current development plan and news-context architecture
- identified the current product gap clearly:
  - MacroLens is better at saying what else moved than what likely happened in the world
- documented a dedicated roadmap for event-grounded context in:
  - `documentation/event_grounded_context_plan.md`
- updated the broader product roadmap so contextual retrieval is now framed as an event-grounded explanation problem rather than only an article-coverage problem
- updated the news-context architecture document so the next direction is explicit:
  - move from generic news retrieval toward event-grounded context

## Problems Encountered

- the existing planning language under-described the real weakness
- "news context quality" was too vague
- the actual issue is that explanations still default to market co-movement too often because real-world event evidence is too sparse or too weakly integrated

## Solution

- turned the problem into a tracked product topic:
  - event-grounded context
- documented the next design shift:
  - anomaly or episode
  - then event context
  - then correlations as supporting structure
- defined a staged build order:
  - episode-level retrieval
  - event/theme extraction
  - explanation-priority rewrite
  - UI separation between drivers, articles, and market relationships

## Next Steps

- implement episode-level context retrieval for clustered anomalies
- change the explanation stack so credible real-world context leads when it exists
- improve the UI language so users can distinguish likely drivers from supporting market evidence

## Additional Work: Episode-Level Context Retrieval

## Goal

Implement the first event-grounded context slice so clustered anomalies retrieve and present context at the episode level rather than only the anomaly timestamp.

## Work Completed

- added episode-aware retrieval to the news-context engine for:
  - `single_dataset_wave`
  - `cross_dataset_episode`
- kept isolated anomalies on anomaly-centered retrieval
- added stored context metadata without changing schema:
  - `retrieval_scope`
  - `timing_relation`
  - `context_window_start`
  - `context_window_end`
- updated the explainer so real-world context now leads when credible context exists
- updated the UI so the event panel now separates:
  - likely real-world drivers
  - supporting articles
  - market relationships
- added regression coverage for:
  - episode-window retrieval
  - retrieval metadata
  - explanation-priority changes
  - API exposure of context metadata

## Problems Encountered

- anomaly-centered retrieval was too weak for clustered monthly and weekly episodes
- the explanation stack was still correlation-first even when contextual evidence existed
- the first UI framing treated all context as one flat news bucket, which hid the intended evidence hierarchy

## Solution

- made retrieval scope explicit and inspectable
- used cluster span as the context window for non-trivial episodes
- kept anomaly-level behavior unchanged for isolated signals
- changed explanation rules so likely real-world context leads when present
- reframed the UI around:
  - likely drivers
  - supporting articles
  - market relationships

## Next Steps

- validate live coverage improvement on real clustered monthly and weekly anomalies
- add event or theme extraction so the likely-driver layer becomes more semantic and less title-first
- continue improving episode-level query quality without over-narrowing retrieval
