# Development Log: March 30, 2026

## Goal

Document the next major product direction for MacroLens, with explicit focus on fixing the weak real-world context layer.

## Work Completed

- reviewed the current development plan and news-context architecture
- identified the current product gap clearly:
  - MacroLens is better at saying what else moved than what likely happened in the world
- documented a dedicated roadmap for event-grounded context in:
  - `documentation/event_grounded_context_plan.md`
- updated the broader product roadmap so contextual retrieval is now framed as an event-grounded explanation problem rather than only an article-coverage problem
- updated the news-context architecture document so the next direction is explicit:
  - move from generic news retrieval toward event-grounded context

## Problems Encountered

- the existing planning language under-described the real weakness
- "news context quality" was too vague
- the actual issue is that explanations still default to market co-movement too often because real-world event evidence is too sparse or too weakly integrated

## Solution

- turned the problem into a tracked product topic:
  - event-grounded context
- documented the next design shift:
  - anomaly or episode
  - then event context
  - then correlations as supporting structure
- defined a staged build order:
  - episode-level retrieval
  - event/theme extraction
  - explanation-priority rewrite
  - UI separation between drivers, articles, and market relationships

## Next Steps

- implement episode-level context retrieval for clustered anomalies
- change the explanation stack so credible real-world context leads when it exists
- improve the UI language so users can distinguish likely drivers from supporting market evidence

## Additional Work: Episode-Level Context Retrieval

## Goal

Implement the first event-grounded context slice so clustered anomalies retrieve and present context at the episode level rather than only the anomaly timestamp.

## Work Completed

- added episode-aware retrieval to the news-context engine for:
  - `single_dataset_wave`
  - `cross_dataset_episode`
- kept isolated anomalies on anomaly-centered retrieval
- added stored context metadata without changing schema:
  - `retrieval_scope`
  - `timing_relation`
  - `context_window_start`
  - `context_window_end`
- updated the explainer so real-world context now leads when credible context exists
- updated the UI so the event panel now separates:
  - likely real-world drivers
  - supporting articles
  - market relationships
- added regression coverage for:
  - episode-window retrieval
  - retrieval metadata
  - explanation-priority changes
  - API exposure of context metadata

## Problems Encountered

- anomaly-centered retrieval was too weak for clustered monthly and weekly episodes
- the explanation stack was still correlation-first even when contextual evidence existed
- the first UI framing treated all context as one flat news bucket, which hid the intended evidence hierarchy

## Solution

- made retrieval scope explicit and inspectable
- used cluster span as the context window for non-trivial episodes
- kept anomaly-level behavior unchanged for isolated signals
- changed explanation rules so likely real-world context leads when present
- reframed the UI around:
  - likely drivers
  - supporting articles
  - market relationships

## Next Steps

- validate live coverage improvement on real clustered monthly and weekly anomalies
- add event or theme extraction so the likely-driver layer becomes more semantic and less title-first
- continue improving episode-level query quality without over-narrowing retrieval

## Additional Work: First-Pass Event Theme Extraction

## Goal

Make likely-driver context more semantic by extracting narrow macro themes from stored context items.

## Work Completed

- added deterministic first-pass event theme extraction inside the news-context pipeline
- stored:
  - `event_themes`
  - `primary_theme`
  in news-context metadata
- exposed those fields through the API
- updated explanations so they can say things like:
  - market stress
  - fed policy
  - inflation
  instead of only repeating a headline
- updated the event panel to show theme badges on likely-driver cards

## Problems Encountered

- the likely-driver section was still too headline-first
- a full semantic extraction system would be too large for this step
- the system needed a useful middle ground between:
  - raw article titles
  - and a heavyweight NLP pipeline

## Solution

- used a deterministic theme taxonomy over:
  - article title
  - query context
  - dataset-specific theme priors
- kept the taxonomy narrow and inspectable
- treated the result as a first-pass cue, not a final event interpretation layer

## Next Steps

- validate whether the extracted themes help on real Fed, CPI, oil, and housing episodes
- improve ranking so the best themed context is more likely to appear in likely drivers
- eventually move from deterministic theme tags toward richer event extraction only if the simpler layer proves useful

## Additional Work: Hybrid Historical Context Fallback

## Goal

Improve real-world context coverage for older and cross-dataset episodes without rewriting the news system.

## Work Completed

- changed the default context mode to `hybrid` so curated historical context is available alongside live retrieval
- expanded curated macro-timeline coverage for:
  - the 2007-2010 financial crisis
  - the 1979-1982 great inflation regime
  - the 2022 Ukraine energy and inflation shock
- changed macro-timeline matching so it can use the full episode dataset set, not only the anomaly's primary series
- changed explanation ranking to prefer curated macro-timeline context over generic live articles when both describe the same episode
- validated the result on live anomalies:
  - March 2022 Fed and WTI anomalies now pick up Ukraine-war context
  - October 2008 WTI now picks up subprime-crisis context

## Problems Encountered

- episode-level retrieval improved structure but still left many historical anomalies with zero context
- the strongest failure mode was not the episode window itself
- the strongest failure mode was sparse fallback coverage for older episodes and for cross-dataset shocks that were not tied cleanly to one dataset keyword
- some anomalies still remain uncovered after this pass, especially early-1990s CPI and 2003 housing

## Solution

- kept GDELT as the recent live provider
- widened curated fallback so it is no longer restricted to household series
- matched timeline entries against cluster dataset overlap, which lets one curated historical event support multiple related datasets in the same episode
- kept the change narrow:
  - no schema migration
  - no new provider
  - no heavy event-extraction layer yet

## Next Steps

- add semantic reranking so the best context item wins more reliably than simple provider ordering
- expand curated coverage for still-empty historical regimes
- inspect whether early-1990s CPI and early-2000s housing need dedicated curated timeline entries or better live query templates

## Additional Work: Structured Historical Event Registry

## Goal

Turn the curated historical context layer into a more explicit event registry so the pipeline can use structured macro-event meaning, not only source titles.

## Work Completed

- extended the curated historical context entries with explicit structured fields:
  - event id
  - event type
  - summary
  - regions
  - themes
  - confidence
- passed those fields through stored `news_context.metadata`
- exposed the new fields through the anomaly API
- updated explanations so curated historical context can use the event summary directly
- updated the frontend likely-driver cards to show the stored historical-event summary when present

## Problems Encountered

- the curated fallback improved coverage, but the app still relied too much on titles
- a source title alone is not the same thing as reusable event evidence
- the system needed a way to say:
  - what the matched event was
  - why it mattered
  - how confident the registry match was

## Solution

- treated the curated historical layer as a small structured event registry
- kept the implementation narrow by reusing the existing `news_context` persistence path
- stored structured event fields in metadata instead of introducing a new table in this slice

## Next Steps

- decide whether the structured event registry should stay metadata-backed or graduate into its own persisted table
- add a better semantic reranker so historical registry context and live articles compete on meaning, not just provider order

## Additional Work: Semantic Context Reranking

## Goal

Improve the quality of the chosen context item so the system selects the most meaningful real-world driver, not just the first acceptable provider result.

## Work Completed

- added a semantic context score to contextual evidence
- the score now blends:
  - timing fit
  - source type
  - theme overlap
  - dataset-title match
  - event specificity
- stored the score in contextual metadata
- updated anomaly-detail reads to order context by score before provider order
- updated explanation primary-context selection to use the score instead of naive provider precedence
- exposed the score through the API for transparency

## Problems Encountered

- once coverage improved, the main failure mode shifted:
  - the system often had context
  - but it still did not always choose the best context
- provider order alone was too blunt
- broad curated context and narrower live context needed to compete on semantics, not just source identity

## Solution

- added a narrow semantic reranker without rewriting the provider system
- kept the scoring inspectable and deterministic
- used the reranker both in repository ordering and explanation selection so the app and the generated explanation stay aligned

## Next Steps

- validate the score against a larger historical benchmark set
- refine theme overlap and specificity scoring where the current event summaries are still too broad
- decide whether the score should eventually become part of a fuller evidence-quality panel

## Additional Work: Targeted Coverage Improvements For Policy, Inflation, And Oil Episodes

## Goal

Improve event-grounded context coverage where the graph-quality snapshot showed the worst gaps:

- `DCOILWTICO`
- `FEDFUNDS`
- `CPIAUCSL`

without broadening retrieval so much that the likely-driver layer becomes noisy again.

## Work Completed

- relaxed clustered query construction for dataset-specific overrides so episode hints no longer over-constrain live retrieval
- expanded policy / inflation / energy query templates for:
  - `CPIAUCSL`
  - `FEDFUNDS`
  - `DCOILWTICO`
- added a curated historical backdrop for the `1964-1978` inflation buildup
- widened the post-pandemic inflation / tightening backdrop to start in mid-`2020` so reopening-era CPI and oil episodes can match it
- added regression tests for:
  - clustered override query construction
  - `1965` Fed-funds historical matching
  - `2020` CPI reopening-era historical matching

## Problems Encountered

- the previous query builder combined dataset overrides with episode hints using a hard `AND`
- that was too restrictive for cross-dataset policy / inflation / energy episodes, especially when the dataset already had a strong specialized query
- once historical coverage widened, some older tests incorrectly assumed only one curated match could exist for a request

## Solution

- kept the dataset-specific override as the primary live query anchor
- changed clustered override behavior so the query can match either:
  - the strong dataset-specific override
  - or a dataset-anchored episode-aware companion query
- updated tests to assert that the intended backdrop is present rather than assuming only one valid curated item exists

## Next Steps

- rerun live context refreshes for representative `DCOILWTICO`, `FEDFUNDS`, and `CPIAUCSL` anomalies
- rerun the graph-quality snapshot after those targeted refreshes
- decide whether the next coverage step should be:
  - a few more curated historical policy entries
  - or stronger live query templates for modern Fed / inflation / oil anomalies
