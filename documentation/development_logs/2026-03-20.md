# Development Log: March 20, 2026

## Goal

Make leading-indicator rankings easier to audit by letting a user compare multiple supporting episodes side by side before drilling into anomaly detail.

## Work Completed

- reviewed the current leading-indicator browser and confirmed the remaining weakness was not missing data, but missing comparative context across supporting episodes
- added a dedicated `LeadingIndicatorCard` render helper in the frontend so the comparison workflow could be implemented without making `App.jsx` even more monolithic
- added per-leader comparison state that lets the user select up to three supporting episodes at once
- reset comparison state on dataset changes so stale comparisons do not leak across contexts
- added a side-by-side comparison panel inside each expanded leading-indicator row
- surfaced comparable episode fields directly in that panel:
  - cluster window
  - target event timestamp
  - matched lag
  - matched correlation
  - target event method
  - target event direction
  - target event severity
  - target cluster peak severity
  - inline cluster-member previews
- added explicit compare / remove controls on each supporting episode row
- kept the workflow inside the current leading-signal panel instead of creating a separate analysis surface
- updated product and architecture documentation so the repo now describes the comparison workflow as implemented state rather than future work
- evaluated live leading-indicator rankings across CPI, Fed funds, mortgage rates, house prices, and real disposable income
- found that mixed-frequency pairs were not uniformly wrong, but daily-to-monthly pairs still looked too comparable to same-frequency pairs under the existing score
- added an explicit `frequency_alignment` term to the leading-indicator score
- exposed the actual source-target frequency pair and frequency-fit value in the UI
- documented the change in a dedicated experiment and ADR
- evaluated the live rankings again and confirmed that another weakness remained: one-cluster leaders could still look too mature when target-cluster count was small
- added an explicit `support_confidence` term to the leading-indicator score
- exposed support confidence directly in the UI so sparse leaders are easier to recognize
- documented the new heuristic in a second experiment and ADR
- reviewed whether `support_confidence` should become continuous and decided to freeze the current stepwise curve for now

## Problems Encountered

- the supporting-cluster browser had already become dense, so adding comparison risked turning it into a second event panel instead of a lightweight research aid
- letting comparison selections persist across dataset changes would have created invalid cross-context UI state
- a single comparison target is not a real comparison, but forcing exactly two selections would have made the tool more rigid than useful
- mixed-frequency relationships are not a simple bug category, so a hard filter would have thrown away useful weekly-to-monthly signals along with weaker daily-to-monthly ones
- a frequency penalty that is too strong would have hidden plausible mortgage-rate and household relationships
- cluster coverage is relative, which meant a `1 of 1` leader could still look deceptively complete
- a hard filter on one-cluster leaders would have hidden useful but immature signals instead of labeling them honestly
- a continuous support-confidence curve would look more precise than the current event corpus justifies

## Solution

- kept the comparison scope narrow: compare selected supporting episodes only within one leading-indicator row
- capped comparison at three episodes so the view remains readable on laptop-sized screens
- kept the comparison metadata-first rather than adding sparklines immediately
- reset comparison selections when the dataset-level view reloads
- reused the existing supporting-episode payload instead of adding more backend surface area
- treated frequency alignment as a smaller score component rather than a filter
- used a simple gap-based heuristic:
  - same frequency: `1.00`
  - adjacent frequencies: `0.85`
  - daily to monthly: `0.65`
- kept the frequency term smaller than coverage, strength, and sign consistency so the score still reflects repeated evidence first
- added a separate absolute-support term so the ranking now expresses both:
  - relative coverage across the target event graph
  - absolute count of distinct supporting clusters
- used a conservative stepwise support-confidence heuristic:
  - `0.20` for one supporting cluster
  - `0.55` for two
  - `0.80` for three
  - `1.00` for four or more
- froze that stepwise curve as the default so the current evidence-and-investigation slice can stabilize

## Next Steps

- use the new comparison workflow against a few hand-checked macro episodes to see whether the current leading-indicator rankings are actually stable across regimes
- decide whether mixed-frequency dataset pairs need additional penalties or filters in the leading-indicator score
- decide whether episode comparison needs small sparklines or whether the current metadata-first view is enough
- improve event clustering and episode quality so the next trust gains come from better event units rather than more heuristic tuning
- continue improving the constellation only after the research workflow is judged trustworthy enough
