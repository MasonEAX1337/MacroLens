# Development Log: March 20, 2026

## Goal

Make leading-indicator rankings easier to audit by letting a user compare multiple supporting episodes side by side before drilling into anomaly detail.

## Work Completed

- reviewed the current leading-indicator browser and confirmed the remaining weakness was not missing data, but missing comparative context across supporting episodes
- added a dedicated `LeadingIndicatorCard` render helper in the frontend so the comparison workflow could be implemented without making `App.jsx` even more monolithic
- added per-leader comparison state that lets the user select up to three supporting episodes at once
- reset comparison state on dataset changes so stale comparisons do not leak across contexts
- added a side-by-side comparison panel inside each expanded leading-indicator row
- surfaced comparable episode fields directly in that panel:
  - cluster window
  - target event timestamp
  - matched lag
  - matched correlation
  - target event method
  - target event direction
  - target event severity
  - target cluster peak severity
  - inline cluster-member previews
- added explicit compare / remove controls on each supporting episode row
- kept the workflow inside the current leading-signal panel instead of creating a separate analysis surface
- updated product and architecture documentation so the repo now describes the comparison workflow as implemented state rather than future work
- evaluated live leading-indicator rankings across CPI, Fed funds, mortgage rates, house prices, and real disposable income
- found that mixed-frequency pairs were not uniformly wrong, but daily-to-monthly pairs still looked too comparable to same-frequency pairs under the existing score
- added an explicit `frequency_alignment` term to the leading-indicator score
- exposed the actual source-target frequency pair and frequency-fit value in the UI
- documented the change in a dedicated experiment and ADR
- evaluated the live rankings again and confirmed that another weakness remained: one-cluster leaders could still look too mature when target-cluster count was small
- added an explicit `support_confidence` term to the leading-indicator score
- exposed support confidence directly in the UI so sparse leaders are easier to recognize
- documented the new heuristic in a second experiment and ADR
- reviewed whether `support_confidence` should become continuous and decided to freeze the current stepwise curve for now
- upgraded event clustering from a single fixed-window grouper to a frequency-aware clusterer
- added persisted episode metadata to `anomaly_clusters`:
  - `span_days`
  - `frequency_mix`
  - `episode_kind`
  - `quality_band`
- wired that metadata through anomaly detail, propagation edges, explanation context, and leading-indicator support episodes
- updated the frontend so the event panel, propagation cards, and supporting-episode browser now surface episode type, quality, and frequency mix directly
- verified the new episode-quality slice against focused tests first and then the full backend suite

## Problems Encountered

- the supporting-cluster browser had already become dense, so adding comparison risked turning it into a second event panel instead of a lightweight research aid
- letting comparison selections persist across dataset changes would have created invalid cross-context UI state
- a single comparison target is not a real comparison, but forcing exactly two selections would have made the tool more rigid than useful
- mixed-frequency relationships are not a simple bug category, so a hard filter would have thrown away useful weekly-to-monthly signals along with weaker daily-to-monthly ones
- a frequency penalty that is too strong would have hidden plausible mortgage-rate and household relationships
- cluster coverage is relative, which meant a `1 of 1` leader could still look deceptively complete
- a hard filter on one-cluster leaders would have hidden useful but immature signals instead of labeling them honestly
- a continuous support-confidence curve would look more precise than the current event corpus justifies
- the original clustering layer only knew timestamps and counts, which meant downstream systems had no way to distinguish a broad cross-dataset episode from an isolated stored signal
- slower monthly series needed wider grouping windows, but widening the window globally would have over-grouped fast daily anomalies even more

## Solution

- kept the comparison scope narrow: compare selected supporting episodes only within one leading-indicator row
- capped comparison at three episodes so the view remains readable on laptop-sized screens
- kept the comparison metadata-first rather than adding sparklines immediately
- reset comparison selections when the dataset-level view reloads
- reused the existing supporting-episode payload instead of adding more backend surface area
- treated frequency alignment as a smaller score component rather than a filter
- used a simple gap-based heuristic:
  - same frequency: `1.00`
  - adjacent frequencies: `0.85`
  - daily to monthly: `0.65`
- kept the frequency term smaller than coverage, strength, and sign consistency so the score still reflects repeated evidence first
- added a separate absolute-support term so the ranking now expresses both:
  - relative coverage across the target event graph
  - absolute count of distinct supporting clusters
- used a conservative stepwise support-confidence heuristic:
  - `0.20` for one supporting cluster
  - `0.55` for two
  - `0.80` for three
  - `1.00` for four or more
- froze that stepwise curve as the default so the current evidence-and-investigation slice can stabilize
- kept the cluster upgrade conservative:
  - daily anomalies still use the base window
  - weekly anomalies expand to a 14-day window
  - monthly anomalies expand to a 35-day window
- tightened mixed-frequency episode formation so daily and monthly anomalies no longer inherit the full monthly gap by default
- replaced the old max-gap bridge with a geometric-mean style pair window:
  - daily to weekly: `10` days
  - weekly to monthly: `22` days
  - daily to monthly: `16` days
- added transparent episode labels instead of inventing a hidden cluster score
- treated episode quality as metadata first, not as a new downstream heuristic multiplier
- added a conservative downstream use of episode quality where it is actually justified:
  - low-quality episodes now reduce propagation strength
  - low-quality episodes now force more cautious explanation framing

## Next Steps

- inspect real clusters to see where mixed-frequency bridging is still too blunt even after the new pair-window rule
- decide whether the current low-quality episode penalty is conservative enough or still too flattering
- decide whether episode comparison needs small sparklines or whether the current metadata-first view is enough
- continue improving the constellation only after the research workflow is judged trustworthy enough

## Additional Work: Zero-Variance Correlation Hardening

### Goal

Eliminate NumPy invalid-divide warnings from the correlation engine without changing the surrounding correlation model.

### Work Completed

- traced the warning path from stored-data refresh into `compute_best_lag_correlation`
- confirmed the warning source was `Series.corr()` on constant percent-change overlaps
- added an explicit zero-variance guard before attempting Pearson correlation
- added regression coverage for constant-window inputs and verified the warning path is now suppressed by design rather than by accident

### Problems Encountered

- the warning did not surface as an exception, which made it easy to ignore even though it represented a real numerical edge case
- the offending condition was not “bad data” globally; it was local overlap windows that became constant after percent-change transformation

### Solution

- added an explicit variance check on both merged overlap series before calling `.corr()`
- skipped correlation attempts when either side had fewer than two distinct overlap values
- kept the fix small and local to the correlation engine instead of adding broader numerical filtering elsewhere

### Next Steps

- rerun the next full stored-data refresh and confirm the warning no longer appears in practice
- inspect whether any remaining correlation edge cases still deserve explicit guards, especially around tiny overlap windows on slower monthly series

## Additional Work: Evidence Refresh Instrumentation And Dataset-Scoped Fast Path

Reduce the iteration cost of `recompute_evidence.py` and stop guessing where refresh time is going.

### Work Completed

- traced the full recompute script and confirmed it always rebuilt:
  - all correlations
  - all news context
  - all explanations
  even when only one dataset changed
- added stage timing instrumentation around:
  - dataset loading
  - anomaly detection

  - correlation rebuild
  - news-context refresh
  - explanation refresh

  - correlations now recompute only for selected datasets
  - news refresh now recomputes only selected-dataset anomalies
  - explanation refresh expands to cluster scope only when reclustering ran
- added scope-regression tests for the new fast-path behavior

### Results

A measured dataset-scoped run:

```text
python scripts/pipeline/recompute_evidence.py --dataset CPIAUCSL --skip-explanations
```

produced:

- load datasets: `0.03s`
- anomaly detection: `0.25s`
- clustering: `1.80s`
- correlation rebuild: `22.66s`
- news-context refresh: `124.02s`
- total: `148.76s`

### Conclusion

The original guess that correlations were the only bottleneck was incomplete.

For dataset-scoped local iteration:

- correlations are still expensive
- news-context refresh is now the dominant measured stage

The new fast path is still a meaningful win because it avoids global downstream rebuilds, but the next obvious bottleneck is now visible rather than hidden.
