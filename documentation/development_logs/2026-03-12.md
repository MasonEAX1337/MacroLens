## Problems Encountered

- the repo initially had only partial documentation and no runnable application structure
- the first backend run failed because `uvicorn` was being executed from the global Python install instead of the project virtual environment
- CoinGecko returned duplicate same-day timestamps with different times, which created chart and analysis distortion
- append-style ingestion left stale rows behind after source normalization logic changed
- correlation computation needed to happen after a full batch ingest, not after each individual dataset, or earlier datasets would be analyzed against incomplete context
- live provider integration needed to preserve the existing evidence boundary instead of letting model output outrun stored facts
- provider expansion increased the need to keep fallback behavior and prompt discipline consistent across backends
- the first Gemini output misread `lag_days` semantics and described the related movement as prior rather than later
- even after fixing the lag direction, Gemini still tended to overstate the relevance of post-event correlations
- the initial dataset story still looked incomplete until a high-recognition equity benchmark was added
- the first news-provider integration hit hard external rate limits and non-JSON rate-limit responses
- the system had grown beyond what unit tests alone could credibly verify
- the first relevance filter pass was too strict and collapsed recent news coverage to zero rows
- the frontend originally allowed only passive inspection, which made provider testing slower than it needed to be
- the richer visual layer raised a new tradeoff: stronger presentation now has a measurable bundle-size cost
- even after chunking, the Three.js core payload still sits just above Vite's large-chunk warning threshold

## Solution

- standardized the main product doc as `MVP.md`
- added documentation rules so the repo records engineering reasoning, not only outcomes
- built a coherent backend, database, ingestion, and frontend scaffold instead of continuing with planning-only artifacts
- ran the backend through the project virtual environment instead of the global interpreter
- normalized CoinGecko daily timestamps to midnight UTC before storage
- changed implemented source loaders to full-refresh writes so corrected normalization can actually clean the database
- moved correlation execution to the batch level so the engine sees the full available dataset set
- added a live OpenAI provider path behind the explanation abstraction and kept the rules-based provider as fallback
- added a Gemini provider path behind the same abstraction instead of creating provider-specific explanation logic
- added explicit lag interpretation to hosted-provider prompts so the model does not have to infer timeline direction from raw integers
- added explicit timing classes and stricter wording guidance so lagging evidence is treated as weak contextual support rather than a likely driver
- extended the FRED dataset registry with the S&P 500 index so the existing ingestion path could cover the original product framing without a new client implementation
- separated news context into its own stored evidence table rather than misusing correlation storage for article retrieval
- added provider throttling and retry handling for GDELT rate-limit responses
- added a separate integration test harness with a dedicated Postgres test database and seeded API assertions
- reintroduced a small timing grace window because GDELT seen timestamps can lag the anomaly date even when the article is still contextually relevant
- added a dedicated `POST /api/v1/anomalies/{id}/regenerate-explanation` path so the UI can rerun explanation generation against stored evidence without re-ingesting data
- kept the new 3D visualization on the existing API contracts instead of adding special-purpose backend endpoints

## Insight

The most important thing learned today is that MacroLens is now constrained less by architecture and more by judgment.

The evidence pipeline exists.

That changes the nature of the work:

- bad future decisions will come from weak interpretation, not missing scaffolding
- the next quality gains will come from better evidence and better interaction, not more folders
- the current weak point is not persistence or API shape, it is the still-transitional explanation layer
- even after live-provider integration, the hard problem is not calling a model API. It is preserving explanation quality and grounding.
- a hosted provider can be technically integrated and still semantically wrong; prompt structure matters as much as transport.
- hosted-provider quality is now primarily a prompt-governance problem, not an API integration problem.
- adding a major dataset after the pipeline exists is cheap if the ingestion boundary is clean; that validates the current provider registry design.
- once basic filters and provenance exist, the next trust problem is not visibility. It is verification.
- adding news helps the product thesis, but it also shifts the next weak point from pure explanation wording to retrieval relevance and citation quality.
- once the integration suite exists, the next trust problem is not whether the app works at all. It is whether the evidence it surfaces is actually good enough to trust.
- once explanation regeneration exists in the UI, the next frontend value is not another button. It is faster comparison and clearer evidence interpretation.
- a visually ambitious frontend feature is only worth keeping if it still behaves like an analysis instrument rather than a demo ornament.
