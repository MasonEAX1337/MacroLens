# Development Log: March 12, 2026

## Goal

Move MacroLens from a documentation-heavy concept into a live, end-to-end vertical slice with real data, persisted evidence, and a working frontend investigation surface.

## Work Completed

- created the top-level documentation system
- finalized the MVP definition and development plan
- scaffolded the FastAPI backend
- wrote the PostgreSQL schema and local environment files
- implemented ingestion for CoinGecko and FRED-backed datasets
- implemented rolling z-score anomaly detection
- implemented lag-aware percent-change correlation scoring
- implemented explanation generation through a provider abstraction
- connected the pipeline to PostgreSQL and ran it against real data
- loaded Bitcoin, CPI, Federal Funds Rate, and WTI into the live database
- verified persisted anomalies, correlations, and explanations through the API
- replaced the frontend placeholder with a live investigation UI
- added CORS support for the frontend
- fixed a source normalization bug in CoinGecko daily timestamps
- changed the current ingestion model to full-refresh replacement so stale source rows do not remain in the dataset

## Evidence Snapshot

At the end of the session, the live database contained:

- 4 datasets
- 12,288 data points
- 83 anomalies
- 54 correlations
- 83 explanations

This matters because the repo is no longer just a planned system. It now has a real evidence graph behind the UI.

## Problems Encountered

- the repo initially had only partial documentation and no runnable application structure
- the first backend run failed because `uvicorn` was being executed from the global Python install instead of the project virtual environment
- CoinGecko returned duplicate same-day timestamps with different times, which created chart and analysis distortion
- append-style ingestion left stale rows behind after source normalization logic changed
- correlation computation needed to happen after a full batch ingest, not after each individual dataset, or earlier datasets would be analyzed against incomplete context

## Solution

- standardized the main product doc as `MVP.md`
- added documentation rules so the repo records engineering reasoning, not only outcomes
- built a coherent backend, database, ingestion, and frontend scaffold instead of continuing with planning-only artifacts
- ran the backend through the project virtual environment instead of the global interpreter
- normalized CoinGecko daily timestamps to midnight UTC before storage
- changed implemented source loaders to full-refresh writes so corrected normalization can actually clean the database
- moved correlation execution to the batch level so the engine sees the full available dataset set

## Insight

The most important thing learned today is that MacroLens is now constrained less by architecture and more by judgment.

The evidence pipeline exists.

That changes the nature of the work:

- bad future decisions will come from weak interpretation, not missing scaffolding
- the next quality gains will come from better evidence and better interaction, not more folders
- the current weak point is not persistence or API shape, it is the still-transitional explanation layer

## Next Steps

- implement a live LLM-backed explanation provider while keeping the rules-based provider as fallback
- add S&P 500 ingestion to complete the initial dataset story
- deepen the frontend with zooming, filtering, and evidence provenance
- add integration tests against the live database rather than relying mostly on unit coverage
