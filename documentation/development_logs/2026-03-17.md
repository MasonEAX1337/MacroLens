# Development Log: March 17, 2026

## Goal

Make the new propagation and change-point layers operationally credible, not just implemented in code.

## Work Completed

- reviewed the first propagation-timeline implementation and identified the main trust problem: a single aggregate score hid too much reasoning
- replaced the opaque propagation score path with a decomposed score model in the backend
- exposed propagation score components for:
  - correlation strength
  - support density
  - temporal alignment
  - target cluster scale
- made the support component more conservative so a single matched link from a single-anomaly source cluster does not look fully supported
- exposed the new score breakdown through anomaly detail
- updated the propagation cards in the frontend to show the score decomposition directly
- added `ruptures`-based `change_point` detection alongside `z_score`
- replaced the original heavy first-pass change-point setup with a faster binary-segmentation configuration that can actually run on the live corpus
- introduced frequency-aware change-point configs for daily, weekly, and monthly datasets
- verified the tuned detector on live datasets:
  - Bitcoin: 1 change point
  - Federal Funds Rate: 14 change points
  - 30-year mortgage rate: 6 change points
- added `scripts/pipeline/recompute_evidence.py` so the evidence graph can be rebuilt from stored data without refetching all sources
- made the refresh workflow resumable by committing news-context and explanation refreshes per anomaly
- added a recent-age cutoff for GDELT so the system stops wasting time on decades-old weak-coverage live-news queries
- hardened GDELT fetches so transport timeouts degrade to empty context instead of aborting the entire refresh
- completed a coordinated live refresh of anomalies, clusters, correlations, explanations, and available news context
- added a leading-indicator discovery layer built on clustered episodes rather than raw anomaly counts
- exposed dataset-level leading indicators through the API and frontend
- added tests for cluster-deduped leading-indicator aggregation and endpoint behavior
- evaluated the first live leading-indicator rankings against CPI, Fed funds, mortgage-rate, and housing examples
- found that several repeated leaders had mixed directional signs across episodes
- added `sign_consistency` and `dominant_direction` to the leading-indicator model and score
- surfaced sign consistency directly in the UI instead of hiding it inside a single ranking number
- added click-through supporting-episode lists to each leading-indicator row so the ranking can be audited against concrete stored anomalies
- upgraded those episode lists into a richer supporting-cluster browser with cluster span, cluster size, target event metadata, and matched lag/correlation context
- added inline cluster-member previews inside that browser so the episode composition is visible before navigation

## Problems Encountered

- the first propagation score was usable for ranking but too compressed for interpretation
- a single number made weak paths look more objective than they really were
- the original support term was too flattering for sparse single-anomaly clusters
- the first change-point implementation was too slow for the real dataset set
- the first full refresh attempt stalled because historical live-news retrieval was trying to treat GDELT like a universal archive
- a single GDELT handshake timeout was enough to abort the whole downstream refresh stage
- some slower monthly household series still produce zero change points under the generic config, which is not automatically wrong but does show that frequency-level defaults are still blunt
- a naive leading-indicator implementation would overcount dense episodes if it treated every target anomaly as an independent vote
- a repeated leader can still be directionally unstable, which made the first ranking look more confident than it should
- a ranked leader without clickable underlying episodes still feels too summary-heavy for a research tool
- a clickable episode list still lacked enough cluster context to judge whether an episode was worth opening
- even after adding cluster context, the browser still made the user infer episode composition without seeing the member anomalies directly

## Solution

- kept the overall score for quick ranking
- added visible sub-scores so the user can inspect why an edge looks strong or weak
- made support density depend on unique supporting source anomalies with a more conservative denominator
- added change-point detection as a second stored method rather than replacing z-score
- switched the live implementation to a faster `binseg` configuration so structural-shift detection is actually runnable inside the product pipeline
- added a dedicated evidence-refresh script so the system can rebuild anomalies and all downstream evidence from stored data
- changed news refresh to use a recent-only live-retrieval policy plus curated historical fallback
- changed long-running downstream stages to commit per anomaly so interrupted refreshes are resumable
- treated provider transport failures as empty-context cases instead of pipeline-fatal exceptions
- aggregated leading indicators at the target-cluster level so repeated support inside one episode does not masquerade as repeated support across episodes
- added sign consistency as a smaller but explicit score component so mixed-sign leaders are penalized instead of silently overrated
- attached the deduped supporting episodes directly to each ranked leader and wired those episodes back into the anomaly detail workflow
- expanded each support row with cluster-level context so the browser can explain why an episode counted before the user navigates away
- attached lightweight cluster-member previews to each support row so the user can see the episode composition without immediately opening anomaly detail

## Next Steps

- expand curated timeline coverage for more historical regimes
- build leading-indicator discovery on top of the current clustered event graph
- evaluate leading-indicator quality against hand-checked macro episodes and tune the ranking if needed
