# Development Log: March 16, 2026

## Goal

Fix the contextual-evidence gap for household macro anomalies without adding another paid news dependency.

## Work Completed

- reviewed the current news-context engine and confirmed the real weakness was provider fit, not database design
- added a curated `macro_timeline` provider alongside the existing GDELT provider
- implemented multi-provider execution in the `news_context` pipeline without changing the database schema
- added household-focused timeline entries for:
  - the subprime mortgage crisis
  - the high-rate Great Inflation period
  - 2020 economic impact payments
  - 2021 third-round economic impact payments
- changed anomaly detail and explanation loading so `macro_timeline` context is surfaced before weaker live retrieval when both exist
- updated provider-status logic so household anomalies report hybrid provider coverage honestly
- tightened hosted-model prompt instructions so curated timeline entries are treated as historical backdrop, not the strongest direct causal evidence
- backfilled live household anomalies and regenerated explanations for representative cases
- updated architecture and README documentation to reflect the new hybrid contextual-evidence model
- added persisted anomaly clustering through new `anomaly_clusters` and `anomaly_cluster_members` tables
- implemented a clustering service that groups adjacent anomalies inside a configurable time window
- wired cluster recomputation into the ingestion pipeline and added a standalone cluster refresh script
- exposed cluster membership through anomaly detail and surfaced it in the frontend event panel
- backfilled the live database so clustering is visible immediately in the running app
- added a first `Propagation Timeline` layer derived from cluster-to-cluster downstream links
- exposed propagation edges through anomaly detail and surfaced them in the frontend event panel
- added conservative edge-strength scoring and click-through navigation to downstream anomalies
- decomposed propagation edge strength into visible sub-scores for correlation, support, timing, and coverage

## Problems Encountered

- GDELT remained structurally weak for household macro topics even after wider windows, deeper fetches, and broader queries
- simply adding a second provider was not enough because the explanation layer initially over-weighted curated timeline items
- if curated context ranked first without prompt discipline, the hosted model would describe it as the strongest relationship rather than background context
- the system still treated anomalies as isolated points even though the next planned product features depend on event-level reasoning
- clustering status had to distinguish attempted providers from actual stored providers or the UI would overstate contextual coverage
- the first propagation pass needed a strict naming boundary or the feature would imply causality too strongly
- propagation summaries can silently mislead if they collapse score direction or over-compress the evidence basis
- a single aggregate propagation score was too easy to over-trust without exposing its components

## Solution

- kept GDELT for live article retrieval
- added `macro_timeline` for selected household regimes where curated history is a better fit than keyword search
- used provider-aware ordering so curated context is visible to users and to the explanation layer
- added explicit prompt guidance telling hosted models to prioritize structured correlations first and use `macro_timeline` only as contextual background
- added a persisted event-clustering layer so the system now has a real macro-event envelope between anomaly detection and future event-chain logic
- built the first downstream propagation view on top of those persisted clusters instead of isolated anomalies
- kept the first version intentionally conservative: downstream only, lagged evidence only, and explicit non-causal wording
- exposed the propagation score components directly so the UI can show why a path looks strong or weak

## Next Steps

- build multi-event event chains on top of the new cluster layer
- expand curated timeline coverage for additional household and policy regimes
- add change-point detection alongside z-score
- decide whether curated timeline entries should be more visually distinct from live article retrieval in the UI
